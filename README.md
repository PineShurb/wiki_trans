# Wiki 翻译工具

一个基于 Tkinter 的桌面小工具，用于把 wiki 类文档翻译为中文，并尽量保持原文结构、术语一致性和既有参考译文风格。

当前版本更适合“续译 + 校订”场景，而不是通用自由翻译器。它支持：

- 本地 Ollama 模型
- OpenAI 兼容的云端接口
- 自定义提示词
- 术语表约束
- 参考文本续译
- 翻译请求与模型原始输出日志记录

## 适用场景

- 翻译 wiki、知识库、技术文档
- 在已有部分译文的基础上继续补译
- 需要统一术语、保留 Markdown / 链接 / 列表格式的文本

## 项目结构

- `translator_ui.py`: 主界面与翻译流程入口
- `ui_pages.py`: 各配置窗口页面
- `translator_gateway.py`: 本地 / 云端模型接口调用
- `config_manager.py`: 配置读取、默认值与保存
- `text_utils.py`: 文本工具与超时解析
- `translator_config.json`: 运行配置文件
- `terminology_template.tsv`: 术语表示例模板
- `logs/`: 每次翻译的日志目录

## 环境要求

- Python 3.10 或更高版本
- Tkinter 可用
- 二选一的模型环境：
  - 本地 Ollama 服务已启动，且目标模型已下载
  - 可访问的 OpenAI 兼容云端接口

本项目当前没有额外的第三方 Python 依赖，主要使用标准库。

如果在 Linux 上启动时报错，提示缺少 Tkinter，可先安装系统包，例如：

```bash
sudo apt install python3-tk
```

## 启动方法

在项目根目录执行：

```bash
python translator_ui.py
```

程序启动后会打开一个三栏界面：

- 左侧：原文输入
- 中间：参考文本
- 右侧：翻译结果

首次启动时，如果 `translator_config.json` 不存在，程序会自动创建默认配置。

## 基本操作流程

### 1. 配置模型

从菜单进入“配置 -> 模型配置”。

可选两种模式：

#### 本地模型

- 模型来源选择“本地模型”
- 本地服务目前仅支持 Ollama
- 填写 Ollama 地址，例如 `http://127.0.0.1:11434`
- 填写模型名称，例如 `qwen2.5:7b`
- 点击“测试模型是否可用”确认服务和模型都可访问
- 点击“保存配置”或“确认并关闭”

#### 云端模型

- 模型来源选择“云端模型”
- 接口地址填写服务根地址，例如 `https://api.openai.com/v1`
- 填写 API Key
- 填写模型名称，例如 `gpt-4o-mini`
- 点击“测试模型是否可用”确认 `/models` 可访问
- 点击“保存配置”或“确认并关闭”

说明：

- 云端接口按 OpenAI 兼容格式调用，请求路径为 `/chat/completions`
- 本地 Ollama 测试会访问 `/api/tags` 和 `/api/generate`

### 2. 配置提示词

从菜单进入“配置 -> 提示词配置”。

默认提示词已经针对 wiki 续译 / 校订做了约束，强调：

- 尽量保持原句顺序
- 保留 Markdown、链接、代码、占位符
- 优先复用参考文本中正确的译法
- 只在发现错误、不一致或缺漏时做最小必要修改

如果你有自己的翻译规范，可以直接在此窗口修改并保存。

### 3. 配置术语表

从菜单进入“配置 -> 术语配置”。

可直接选择现有 TSV 文件，也可以先使用仓库内的 `terminology_template.tsv` 作为模板。

术语表要求：

- 使用制表符分隔
- 每行至少两列：原词、译词
- 第三列可选，作为备注
- 以 `#` 开头的行为注释，会被忽略
- 第一行如果是表头，也会被自动跳过

示例：

```tsv
source_term	target_term	note
CPU	中央处理器	硬件术语，全文统一
knowledge base	知识库	不要翻成“知识基础”
```

路径说明：

- 可以填写绝对路径
- 也可以填写相对路径
- 相对路径会相对于项目目录解析

### 4. 输入待翻译内容

- 把原文粘贴到左侧“原文输入”
- 如果已有部分译文或历史译文，可粘贴到中间“参考文本”
- 点击“开始翻译”

翻译完成后，右侧会显示模型返回的结果。

### 5. 清理界面内容

菜单“工具”中提供：

- 清空输入
- 清空输出
- 全部清空

## 配置文件与日志

### 配置文件

程序会把模型配置、提示词、术语表路径和字体设置保存到 `translator_config.json`。

注意：

- 云端 API Key 会明文保存在该文件中
- 这个文件不适合直接分享或提交到公开仓库

### 日志文件

每次点击“开始翻译”时，程序会在 `logs/` 目录下生成一个新日志文件，例如：

```text
logs/trans_20260519_153000_123456.log
```

日志内容包括：

- 发送给模型的完整提示词
- 当前参考文本
- 当前命中的术语要求
- 模型原始输出

这对排查问题很有帮助，但也意味着日志里可能包含敏感内容。

## 注意事项

### 1. 当前版本一次只发送一整段输入

当前代码会把左侧输入框里的全部内容作为一次翻译请求发送给模型，而不是按段落或句子自动切块。

这意味着：

- 超长文本更容易触发上下文超限
- 云端请求更容易超时
- 本地模型在显存不足时更容易变慢或失败

建议先用较小批次手动分段翻译，确认质量后再继续。

### 2. 参考文本会整体拼接进提示词

只要中间“参考文本”不为空，程序就会把整段参考文本一并发给模型。

这有助于续译，但也会带来两个影响：

- prompt 变长，增加上下文压力
- 如果参考文本本身有错误，模型可能继续沿用错误风格

建议只放入真正有参考价值的已译内容。

### 3. 术语表是按文本命中后再注入提示词

程序会从当前待翻译文本中查找是否包含术语表里的原词；只有命中的术语才会被拼进提示词。

注意：

- 匹配方式是简单的大小写不敏感子串匹配
- 不会做词形还原、同义词扩展或模糊匹配
- 如果术语表里的原词和原文写法差异较大，可能不会命中

### 4. 超时设置目前主要用于“测试连接”

界面中的超时参数会用于模型可用性测试。实际翻译请求阶段当前没有显式传入该超时值。

因此，如果模型接口响应很慢，界面可能会等待较久。

### 5. 翻译过程中界面可能短暂卡住

翻译请求是在界面线程中直接发起的。模型响应较慢时，窗口可能出现短暂无响应，这是当前实现的正常现象。

### 6. 翻译结果不会自动保存到文件

右侧输出框中的内容目前只显示在界面里，不会自动导出到单独文件。

如果结果需要留存，请手动复制保存。

### 7. 输出配置页面尚未实现

菜单里虽然有“输出配置”，但当前点击后只会弹出待实现提示。

### 8. 术语表格式错误会直接中止本次翻译

如果术语表路径无效、文件不存在、制表符格式不正确，程序会弹窗提示并停止本次翻译。

建议先用模板文件验证格式，再替换为正式术语表。

### 9. 明文配置与日志需要妥善保管

以下内容都可能包含敏感信息：

- `translator_config.json` 中的 API Key
- `logs/` 中记录的原文、参考文本、提示词和模型输出

如果仓库需要同步到 Git，请优先确认这些文件是否应该加入忽略规则。

## 常见问题

### 本地模型测试失败

优先检查：

- Ollama 是否已启动
- 地址是否填写正确
- 模型是否已经拉取到本地

可先在终端自行确认模型列表，例如：

```bash
ollama list
```

### 云端模型测试失败

优先检查：

- 接口地址是否填写为根地址，而不是完整补到某个具体接口
- API Key 是否有效
- 目标服务是否兼容 `/models` 和 `/chat/completions`
- 模型名称是否和服务端返回列表一致

### 术语表没有生效

优先检查：

- 文件是否为 TSV，而不是逗号分隔 CSV
- 原词是否真的出现在待翻译文本中
- 原词拼写、大小写、连字符、复数形式是否与原文一致或足够接近

## 建议使用方式

为了得到更稳定的结果，建议按下面顺序使用：

1. 先配置模型并通过连通性测试
2. 再确认提示词是否符合当前 wiki 风格
3. 如需统一术语，先准备 TSV 术语表
4. 把长文拆成较小片段逐段翻译
5. 每段翻译后检查日志与输出，再继续下一段

## 已知限制

- 当前 UI 只有单次“开始翻译”，没有暂停、取消、断点续译
- 本地模型选项目前只实现了 Ollama
- 没有自动保存输出文件功能
- 没有自动分块、自动重试、自动回滚机制
- 没有对敏感信息做脱敏日志处理

如果后续继续开发，优先值得补的方向通常是：自动切块、翻译超时控制、输出导出、异步请求与更细粒度的错误提示。
